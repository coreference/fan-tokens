## Single-file vs. split ‚Äî decision

**Keep `nfl-fan-wallet-prototype.html` as a single file.** Reasons:
- The project instructions explicitly mandate "interactive single-file HTML/JS Artifacts" ‚Äî this is a Claude-artifact / Figma-import deliverable, not a deployed app with a build pipeline. Splitting would break that delivery model for no benefit.
- It's already organized internally like a small app (design tokens ‚Üí state ‚Üí component helpers ‚Üí per-screen registry ‚Üí flow/scenario config ‚Üí router/render ‚Üí boot), so the lack of file boundaries isn't causing disorganization ‚Äî see "Internal architecture" below.
- At ~1,800 lines / 137KB it's still comfortably editable with targeted greps/edits; it hasn't hit a size where single-file becomes a liability.
- Splitting would require introducing a bundler/build step, which conflicts with "no build step" simplicity and the Figma-import constraint (no module loading at runtime).

If it keeps growing (e.g. another 1,500+ lines, or multiple people editing concurrently), reconsider ‚Äî but that's not the current state.
