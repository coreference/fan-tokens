## Single-file vs. split — decision

**Keep `nfl-fan-wallet-prototype.html` as a single file.** Reasons:
- The project instructions explicitly mandate "interactive single-file HTML/JS Artifacts" — this is a Claude-artifact / Figma-import deliverable, not a deployed app with a build pipeline. Splitting would break that delivery model for no benefit.
- It's already organized internally like a small app (design tokens → state → component helpers → per-screen registry → flow/scenario config → router/render → boot), so the lack of file boundaries isn't causing disorganization — see "Internal architecture" below.
- At ~1,800 lines / 137KB it's still comfortably editable with targeted greps/edits; it hasn't hit a size where single-file becomes a liability.
- Splitting would require introducing a bundler/build step, which conflicts with "no build step" simplicity and the Figma-import constraint (no module loading at runtime).

If it keeps growing (e.g. another 1,500+ lines, or multiple people editing concurrently), reconsider — but that's not the current state.

## Internal architecture of the HTML file

**No line numbers below — the file is grep-navigated.** Every section is marked with a banner comment (`/* ====... */` framing a title line) in the CSS and in the `<script>`; grep the exact banner text or identifier quoted in each bullet to jump straight there regardless of how much the file has grown or shrunk.

- **`<style>` block** — grep `DESIGN TOKENS` for CSS custom-property tokens in `:root` (studio/outside-phone chrome), then per-skin overrides on `.phone[data-skin="panthers|dolphins|boost|patriots|eagles|niners"]` (league skin = the unadorned `.phone` default; look for the `Task 1 — pilot-team skins` comment just above the overrides). Grep `STUDIO LAYOUT` for the outer studio chrome, `PHONE FRAME` for the phone bezel + in-phone component classes (`.card`, `.btn`, `.chip`, `.qchip`, `.medallion`, etc), `ANNOTATION PANEL` for the right-rail panel styles, `TASK 3 — MANUAL SKIN TOGGLE` and `TASK 2 — SCENARIO SELECTOR (SCREEN 0)` for those studio-only controls.
- **Global state & static config** — grep `GLOBAL STATE` for `var state = {…}`, the single mutable state object (current screen, history stack, skin, wallet/points/boost/raffle/vote data, acquisition fields under the `acquisition / onboarding (Task 1)` and `scenario selector + manual skin (Task 2 / Task 3)` comments, edge-case flags under `state.e.*`). Immediately below it: `var TEAMS`, `var SKIN_OF`, `function teamSkin`, `var PACKS`, `var I` (icon SVG strings) — static lookup tables. Team-badge helpers `teamSq()`, `teamHeaderFull()`, `teamHeaderRight()`, `currentTeamId()` live in this same region.
- **Reusable phone components** — grep `REUSABLE PHONE COMPONENTS` for `statusBar()`, `head()`, `btn()`, `tabbar()`, `pill()`, `openSheet()`/`openModal()`/`closeOverlay()`, `toast()` — render helpers used across every screen. Acquisition-specific render helpers (`acqT()`, `rewardHero()`, `qrBox()`) sit just above this banner, near the reward-framed screens that use them.
- **Screen registry** — grep `^var S = {}` for the start of the screen registry (banner text: `SCREENS  — each: flow, phase, step, skin, story, ac[], eng, ctas`). Individual screens are declared as `S.<name> = {…}` (e.g. `S.splash`, `S.profile`, `S.otp`, `S.rewardLanding`, `S.dashboard`, `S.manage`) — grep `^S\.` to list all of them. Each has `flow`, `phase`, `step`, `skinTag`, `story` (user story text), `ac[]` (acceptance criteria), `eng` (engineering note), `ctas[]`, `prev`/`next`, optional `variants[]` (edge/fallback state buttons), and a `body()` function returning the screen's HTML. This is the main place screen content/copy/logic gets edited.
- **Flow rail** — grep `FLOW RAIL` for `var FLOWS = […]`, the top flow-rail tab definitions (acquisition/onboarding/purchase/team/pledge/points/quests/rewards/boost/manage), each mapped to an entry screen id, plus `renderRail()` and `goFlow()` (seeds prerequisite state — e.g. `acqTeam`, `primaryTeam` via `seedPledged()` — so a flow tab renders sane data when entered mid-flow).
- **Router + render core** — grep `ROUTER + RENDER` for `phoneHTML()` (renders one screen into the phone frame, skin-aware), `render()` (full re-render: rail, skin toggle, prev-phone, current phone, annotation panel), `panelHTML()` (right-side annotation panel from a screen's metadata), `SPINE`/`ACQ_SPINE`/`SPINE_PREV` (ordered happy-path screen-id lists and prev-screen overrides used by the panel's "Next happy step" button and back navigation), `happyPrev()`, `advance()`/`go()`/`back()`, `resetTransient()`, `ev()` (edge-flag toggle helper).
- **Interaction handlers** — grep `INTERACTION HANDLERS` for the start of one function per interactive action: registration/OTP (`regSubmit`, `otpKey`, `otpVerify`, `otpResend`, `signinAs`), purchase (`setQty`, `pickMethod`, `reviewGuard`, `openPay`, `confirmPay`, `afterPurchase`), location/team (`allowLocation`, `deny`, `chooseTeam`, `searchTeam`, `confirmManual`), pledge/quests/raffle/voting (`confirmPledge`, `seedPledged`, `questOpen`, `raffleN`, `raffleConfirm`, `voteSel`, `voteConfirm`, `shareSheet`), boost/manage (`startBoost`, `allocSet`, `setPledgePct`, `saveAlloc`, `changeTeamModal`, `switchToDolphins`), and the acquisition handlers (`acqHow`, `acqLogin`, `setPack`, `acqPay`, `acqConfirmPay`, `acqAgeGate`, `acqHandoff`, `acqDash`, `acqReauth`, `goWaitlist`, `joinWaitlist`, `acqContinueNeutral`) — all mutate `state` then call `render()`.
- **Scenario selector + manual skin** — grep `TASK 2 — SCENARIO SELECTOR (SCREEN 0)  ·  TASK 3 — MANUAL SKIN` for this whole region. Contains `S.scenario0` (router marker only, rendered by the studio not the phone), `var SCEN` (the three-Layer acquisition scenario model L1/L2/L3, matching the "Acquisition Scenarios" doc — each layer has `scenarios[]` with id/label/route/core/best/target), `activeTeamName()`/`activeTeamAbbr()`/`activeTeamColor()`, `SKIN_TEAM`, `SKIN_LIST`, `renderSkinToggle()`, `setManualSkin()` (Task 3 — manual pilot-team skin preview independent of acquisition flow), and `scenFind()`, `renderScenarioHub()`, `scenHover()`, `scenReset()`, `launchScenario()`, `phoneOff()`, `scenarioPanel()` (Task 2 — scenario hub rendering).
- **Coverage / assumptions** — grep `COVERAGE / ASSUMPTIONS` for `var COV = {…}` — **the built-in, self-documenting Source Truth Table.** `COV.conf` (built & confirmed, with doc § references), `COV.assm` (assumptions made), `COV.need` (open questions blocking further build-out), `COV.excl` (explicitly out of scope). `renderCov()` renders this into a coverage grid in the studio UI. **Check `COV.need` before any session that extends scope — it's the live list of unresolved product questions**, e.g.: boost rule trigger point, multi-team split shipping in beta, final tier names beyond Bronze→Platinum, referral qualification condition, points-earn formula, age-gate flow for in-stadium beer voucher, per-team reward names/values, non-Panthers rich-dashboard parametrization.
- **Boot** — grep `^/\*  BOOT` for the trailing `renderCov(); render();` boot call at the end of the `<script>` block.

Build history breadcrumbs in the code: "Task 1" = pilot-team skin tokens + acquisition pack/skin config, "Task 2" = scenario selector/hub, "Task 3" = manual skin toggle independent of acquisition state. Use this numbering if the user refers to "Task N" work.

## Pilot teams (5, with full skins)
Carolina Panthers, Miami Dolphins, Philadelphia Eagles, San Francisco 49ers, New England Patriots — each has dedicated `--accent`/`--accent2`/`--grad1`/`--grad2` tokens in the CSS and an entry in `SKIN_OF`/`SKIN_TEAM`/`SKIN_LIST`. Atlanta Falcons and New Orleans Saints exist only in `TEAMS` (search-picker realism, no dedicated skin — see `COV.assm`). All other teams route to the league-neutral skin or the waitlist screen (`S.waitlist`, unsupported-team path).
